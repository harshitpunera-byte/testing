===============================================================================
1. ONE-LINE SUMMARY
===============================================================================

This project is a FastAPI + React tender-resume matching system that uploads PDF
documents, extracts page-aware text, chunks and embeds that text, stores
documents and vectors in PostgreSQL with pgvector, extracts structured tender
and resume fields, and then supports QA, structured resume search, and
deterministic resume matching with reasoning summaries.


===============================================================================
2. 30-SECOND ANSWER
===============================================================================

- I built a tender-resume matching RAG system with a React frontend and FastAPI
  backend.
- Users upload one tender and multiple resumes as PDFs.
- The backend validates files, deduplicates them with SHA256, extracts page-wise
  text through multiple PDF fallbacks, cleans and chunks that text, and stores
  both relational data and vector data in PostgreSQL with pgvector.
- It extracts structured tender and resume data using heuristics plus an
  Ollama-backed JSON extraction layer.
- From there the app can answer questions over uploaded documents, run
  structured resume search, or rank resumes against the uploaded tender using
  deterministic scoring plus a lightweight reasoning summary.


===============================================================================
3. 2-MINUTE ANSWER
===============================================================================

- The frontend is React with Vite and Tailwind. It uploads tenders and resumes,
  sends queries, and shows extracted JSON, evidence, and ranked profiles.
- The backend is FastAPI. Upload routes call a shared ingestion pipeline in
  `app/services/document_ingestion.py`.
- That pipeline validates PDFs, computes SHA256 for duplicate detection, saves
  files under `uploads/`, extracts page-aware text with a fallback chain
  including PyMuPDF4LLM, PyMuPDF, pdfplumber, Docling, and OCR, cleans page
  noise, groups text into semantic sections, splits it into overlapping chunks,
  and generates embeddings with `BAAI/bge-small-en` or a deterministic hash
  fallback.
- Chunks, structured JSON, normalized resume rows, and vector embeddings are
  stored in PostgreSQL through SQLAlchemy and pgvector.
- After storage, the system runs structured extraction. Heuristics run first,
  then Ollama-backed schema-shaped extraction fills gaps, and evidence is mapped
  back to pages and sections.
- On query, the backend classifies intent. QA uses hybrid retrieval over stored
  chunks and asks Ollama for a context-grounded answer. Matching loads tender
  requirements, retrieves candidate resumes, scores them deterministically, and
  uses a small LangGraph step to add explanations and a shortlist summary.

- create `documents` row
- extract pages

- section + chunk
- embed
- persist `document_chunks`
- structured extraction
- persist normalized profile rows for resumes
- persist evidence
- update `documents` to `stored`

6.3 QA flow
- intent classified as `qa`
- hybrid retrieval over `document_chunks`
- structured context loaded from `documents`
- prompt built for Ollama
- answer returned with sources

6.4 Matching flow
- intent classified as `matching`
- resolve tender scope
- load tender requirements
- derive better resume search query
- resolve candidate resumes
- load normalized/extracted resume profile
- repair missing candidate name/role if needed
- deterministic score
- LangGraph reasoning summary

6.5 Structured search flow
- parse natural-language constraints
- apply filters on normalized resume tables
- optionally rerank with semantic pgvector search
- return paginated results

9. CORE TECHNICAL BUILDING BLOCKS
===============================================================================

Validation
- extension check
- size check
- content-type check
- `%PDF` signature check
- readable-page check using PyMuPDF

Deduplication
- SHA256 over raw uploaded bytes
- duplicate key is `document_type + file_hash`
- duplicate uploads reuse stored structured data and persisted chunks

Extraction order
1. PyMuPDF4LLM
2. PyMuPDF
3. pdfplumber
4. Docling
5. OCR via Tesseract

Chunking
- 800-word chunk size
- 150-word overlap
- carries section and page metadata

Embeddings
- primary model: `BAAI/bge-small-en`
- dimension: `384`
- fallback: deterministic hash embedding

Retrieval
- semantic retrieval via pgvector cosine distance
- lexical retrieval via token overlap
- hybrid retrieval via reciprocal rank fusion

Scoring
- required skills: 70
- preferred skills: 10
- role: 10
- domain: 10
- experience: 10

13. GOOD INTERVIEW ANSWERS
===============================================================================

Q. Why did you use RAG here?
A.
- Tender and resume PDFs are long, noisy, and semi-structured.
- Retrieval lets the system use only relevant chunks instead of sending entire
  files to the model every time.

Q. Why not let the LLM do everything?
A.
- Heuristics are faster, cheaper, and more stable for fields like names,
  experience, skills, and role labels.
- The LLM is used as a second pass for schema-shaped extraction and reasoning.

Q. Why PostgreSQL + pgvector?
A.
- It keeps structured relational data and vector search in one primary backend.
- I can store document metadata, normalized resume profiles, and embeddings in a
  single database instead of splitting source-of-truth across local files.

Q. How do you handle duplicate uploads?
A.
- I hash the uploaded bytes with SHA256.
- For the same document type, if the hash already exists and the previous record
  is usable, I reuse the stored parse result instead of reprocessing the file.

Q. How do you rank resumes?
A.
- Ranking is deterministic, not LLM-scored.
- Required skills contribute most of the score, then preferred skills, role,
  domain, and experience.

Q. What does LangGraph do here?
A.
- It adds human-readable explanations and a shortlist summary after the
  deterministic scoring step.
