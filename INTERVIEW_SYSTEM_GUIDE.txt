===============================================================================
1. ONE-LINE SUMMARY
===============================================================================

This project is a FastAPI + React system that uploads tender PDFs and resume PDFs,
extracts page-aware text from them, cleans and chunks that text, embeds chunks
into FAISS, stores document/chunk metadata in SQLite through SQLAlchemy,
extracts structured fields from both tender and resumes using heuristics plus
Ollama-backed JSON extraction, and then either:

1. answers questions over uploaded documents, or
2. ranks resumes against the uploaded tender using deterministic scoring.


===============================================================================
2. BEST INTERVIEW ANSWERS
===============================================================================

30-second answer
- I built a tender-resume matching RAG system. Users upload one tender and many
  resumes as PDFs. The backend validates and deduplicates files using SHA256,
  extracts page-wise text with multiple PDF fallbacks, cleans headers and
  footers, creates semantic chunks, stores embeddings in FAISS, persists
  structured outputs and chunks in SQLite, and then uses retrieval plus
  deterministic scoring to shortlist candidates. The UI lets users upload files,
  ask questions, and inspect extracted JSON and ranked profiles.

2-minute answer
- The backend is FastAPI. Upload routes call a shared ingestion service. That
  service validates PDF uploads, computes SHA256 for deduplication, saves files
  locally, extracts page-level text using PyMuPDF4LLM or other fallbacks,
  cleans repeated headers/footers, groups text into semantic sections, splits
  those sections into overlapping chunks, embeds chunks with
  `BAAI/bge-small-en` or a hash fallback, stores them in FAISS, and also stores
  document and chunk metadata in SQLite through SQLAlchemy.
- After chunk storage, the system runs hybrid extraction. First it applies
  deterministic heuristics to pull fields like role, domain, skills, and
  experience. Then it calls Ollama to get schema-shaped JSON and merges both
  outputs.
- When the user sends a query, the system first classifies intent. If it is a
  normal question, it does hybrid retrieval over tender and/or resume chunks and
  sends that context to the LLM for an answer. If it is a matching request, it
  extracts or loads tender requirements, finds candidate resumes, extracts their
  profile data, repairs missing candidate name and role if needed, computes a
  deterministic score, and runs a small LangGraph reasoning step to add
  explanations and a shortlist summary.

What makes this project stronger than a basic demo
- It is not only a chatbot. It has upload, validation, deduplication,
  persistence, retrieval, structured extraction, evidence mapping, scoring, and
  UI inspection.
- It does not depend only on LLM output. It uses deterministic heuristics before
  and after the LLM.
- It is page-aware and stores chunk metadata, so answers and evidence can point
  back to pages and sections.

5. CORE TECHNICAL BUILDING BLOCKS
===============================================================================

5.1 PDF validation
- File must have a filename.
- File extension must end with `.pdf`.
- File cannot be empty.
- File size must be <= `MAX_UPLOAD_FILE_SIZE_MB` from env.
- Content type must be in allowed PDF types.
- File bytes must start with `%PDF`.
- If PyMuPDF exists, the file is opened to confirm readable page count.


5.2 Deduplication
- SHA256 is computed over raw uploaded bytes.
- Deduplication is per `document_type` + `file_hash`.
- If the same resume or tender is uploaded again and the old record is not
  failed, the app does not re-parse it.
- It updates the existing DB row, reuses stored structured data, reuses persisted
  chunks, renames chunk metadata filename if needed, deletes current FAISS files,
  and lets the vector index rebuild from the DB on next search.


5.3 PDF extraction backend order

Primary family
1. PyMuPDF4LLM page-chunk extraction
2. Plain PyMuPDF extraction

Fallback pipeline if PyMuPDF result is not meaningful
3. pdfplumber
4. Docling
5. OCR using PyMuPDF rasterization + pytesseract

Meaningful-text check
- Requires more than a minimum character count.
- Removes picture-placeholder noise.
- Strips non-letter noise.
- Requires enough meaningful tokens.


5.4 Cleaning
- Detects repeated first lines across pages as likely headers.
- Detects repeated last lines across pages as likely footers.
- Removes page-number-only lines.
- Keeps per-page structure instead of flattening immediately.


5.5 Semantic structuring
- Each page is scanned line by line.
- The code looks for section markers.
- Tender sections include:
  eligibility, qualifications, experience, responsibilities, personnel,
  commercial.
- Resume sections include:
  summary, skills, experience, projects, education, certifications.
- If no marker is found, the block stays under `general`.

Important real behavior
- Blocks are flushed at the end of each page, so sections do not naturally span
  across multiple pages in a single block.


5.6 Chunking
- Word-based chunking, not token-based chunking.
- Default size = 800 words.
- Overlap = 150 words.
- Chunk metadata stores:
  filename, chunk_id, document_id, document_type, section, page_start, page_end.


5.7 Embeddings
- Primary model: `BAAI/bge-small-en`
- Dimension: 384
- Loaded with `local_files_only=True`

Fallback behavior
- If local model is unavailable, the app uses a deterministic SHA256-based hash
  embedding.
- That keeps the system functional even without the transformer model.


5.8 Vector storage
- FAISS index type: `IndexFlatL2`
- Separate index names:
  `tender`
  `resume`
- Metadata also saved to pickle files.
- If the FAISS files are missing, the system can rebuild them from the SQL DB
  chunk records.


5.9 Hybrid retrieval
- Used mainly in QA flow.
- Combines:
  1. semantic vector search
  2. lexical keyword overlap search
- Fusion method:
  Reciprocal Rank Fusion (RRF), `RRF_K = 60`

Lexical scoring includes
- token overlap ratio
- direct substring bonus
- contiguous-overlap bonus


5.10 Structured extraction strategy
- Tender and resume extraction both follow:
  1. heuristic extraction first
  2. LLM JSON extraction second
  3. merge both results

Why this is useful
- Heuristics keep the output stable and cheap for obvious cases.
- LLM fills in fields heuristics miss.
- Schema validation reduces hallucinated shape errors.


5.11 Evidence mapping
- For every structured field, the system searches chunks for the best textual
  support.
- It scores each chunk by exact phrase match or token overlap.
- Then it stores:
  value, source_text snippet, page, section, confidence.


5.12 Deterministic scoring
- Required skill match: up to 70
- Preferred skill match: up to 10
- Role match: 10
- Domain match: 10
- Experience match: 10
- Final score capped at 100

Verdict rules
- `Highly Suitable` if score >= 80 and experience matches
- `Partially Suitable` if score >= 50
- `Low Suitable` otherwise

7. DETAILED BEHAVIOR OF THE MOST IMPORTANT FILES
===============================================================================

7.1 `app/services/document_ingestion.py`

Why it matters
- This is the upload backbone of the whole system.

What it does in order
1. Validates `document_type` is either `tender` or `resume`.
2. Reads uploaded file bytes from FastAPI `UploadFile`.
3. Validates PDF structure.
4. Computes SHA256 hash.
5. Checks duplicate document for same type.
6. If duplicate and existing record is usable:
   - updates existing record status
   - fetches persisted chunks
   - renames stored chunk metadata filename
   - invalidates FAISS so it can rebuild with updated metadata
   - returns existing structured data/evidence without reprocessing
7. If not duplicate:
   - chooses upload directory by document type
   - saves file with timestamp-prefixed sanitized filename
   - creates `Document` row with status `processing`
8. Extracts page-aware text from the PDF.
9. Cleans pages.
10. Flattens pages to full text.
11. Rejects file if no readable text remains.
12. Builds chunk records with page and section metadata.
13. Stores chunk embeddings in FAISS.
14. Replaces persisted chunk rows in SQL DB.
15. Runs structured extraction:
   - tender -> `extract_tender_requirements`
   - resume -> `extract_resume_data`
16. Builds evidence map from structured data and chunks.
17. Updates `Document` row with:
   - status = `stored`
   - extraction backend
   - page count
   - structured_data
   - evidence_map
18. Returns a UI-friendly payload.


7.2 `app/services/query_service.py`

Why it matters
- `/match/` always enters here first.
- This file is the traffic controller between QA and matching.

Important concepts
- Active scope:
  If the UI sends `tender_document_id`, `resume_document_ids`, or
  `restrict_to_active_uploads=true`, the backend tries to answer only from those
  uploaded files.
- Latest-document fallback:
  If no active IDs are provided, it falls back to the latest stored tender or
  resume where appropriate.

How it decides mode
- Calls `classify_query_intent()`.
- Matching mode only happens when:
  1. tender data exists
  2. resume data exists
  3. query contains matching-style keywords
- Otherwise it falls into QA mode with scope tender/resume/both.

How QA works
- Search results are gathered with hybrid retrieval.
- It mixes structured context from DB with retrieved chunk text.
- It builds a strict prompt telling the LLM to answer only from provided context.
- If LLM fails, it returns a fallback answer built from chunk snippets.


7.3 `app/services/matching_service.py`

Why it matters
- This is the heart of ranking and shortlisting.

Matching steps
1. Resolve tender scope from explicit document ID or vector search.
2. Load tender chunks from DB first, FAISS metadata second.
3. Load existing tender structured data from DB or extract it fresh.
4. Build a better resume search query from tender role/domain/skills/exp.
5. Retrieve resumes:
   - if active resume IDs were provided, use those documents directly
   - else vector search top resumes
6. For each resume:
   - load persisted chunks
   - load or extract structured profile
   - repair missing candidate name / role if needed
   - compute deterministic score
   - build excerpt
7. Sort by score and experience match.
8. Run LangGraph reasoning for explanations and summary.

Scoring logic details
- Skill matching uses phrase normalization and overlap, not only exact equality.
- Experience is integer-only when scoring.
- Role/domain matching are text comparisons with normalized token overlap.


7.4 `app/extraction/resume_extractor.py`

Why it matters
- This is the most heuristic-heavy file in the repo.

What it extracts
- candidate_name
- role
- domain
- skills
- experience
- qualifications
- projects

How name extraction works
- Looks near labels like:
  `Name of Staff`, `Name of Candidate`, `Candidate Name`
- Tries inline label extraction.
- Tries extraction between labels.
- Scores nearby lines based on:
  - label proximity
  - whether address follows
  - whether date appears before
  - whether line appears near top of file
- Sanitizes against a large noise-token list.

How role extraction works
- Looks for labels like:
  `Proposed Position`, `Position`, `Role`, `Profession`
- If not found, tries summary patterns like:
  `X with 10 years of experience`
- If still not found, inspects top header lines for likely job-title words.

How experience extraction works
- Direct regex for `X years of experience`
- Label-based extraction such as `Years with Firm/Entity`
- If needed, computes years by parsing date ranges across resume text

How projects extraction works
- Uses long regex patterns for common infrastructure project wording
- Filters obvious noise snippets
- Deduplicates similar project strings

Merge strategy
- Heuristic values are preferred.
- LLM values fill gaps.


7.5 `app/extraction/tender_extractor.py`

What it extracts
- role
- domain
- skills_required
- preferred_skills
- experience_required
- qualifications
- responsibilities

How it works
- Uses domain keywords like highway, bridge, civil, python, machine learning.
- Uses tender-title heuristics to infer role.
- Uses skill patterns to infer required skills.
- Uses personnel-related experience patterns to detect years.
- Uses preferred markers like `preferred`, `preferably`, `desirable`.
- Uses coarse qualification and responsibility detection from tender terms.

Important actual behavior
- In the final merge, `preferred_skills` and `experience_required` come only from
  heuristics. The code does not currently fall back to LLM values for those two
  fields if heuristics miss them.


7.6 `app/llm/provider.py`

What it does
- Loads env vars.
- Builds Ollama `/api/chat` URL.
- Chooses model per task:
  extraction, reasoning, or default.
- Sends either:
  - schema-guided JSON extraction request, or
  - plain text answer request.

Fallback behavior
- For JSON extraction, if Ollama is unavailable, returns a schema-shaped default
  JSON object instead of failing hard.
- For text answering, if Ollama is unavailable, returns an empty string and lets
  caller build a fallback answer.

Important actual behavior
- The env has `LLM_PROVIDER`, but current provider code effectively supports only
  Ollama right now.
- README and PRD mention Gemini, but there is no active Gemini transport code in
  `app/llm/provider.py`.


7.7 `app/rag/vector_store.py`

What it does
- Embeds text.
- Saves FAISS indexes.
- Saves metadata pickles.
- Loads indexes lazily.
- Rebuilds missing FAISS files from SQL DB chunk records.
- Supports semantic search and hybrid semantic+lexical search.
- Fetches all chunks for a given document or a local window around a chunk ID.

Important design point
- SQL DB is close to the source of truth for chunk metadata.
- FAISS is the fast retrieval layer and can be regenerated.

12. INTERVIEW QUESTIONS AND GOOD ANSWERS
===============================================================================

Q1. Why did you use RAG here?
A.
- Tender and resume PDFs are long, semi-structured documents.
- I needed retrieval so the system could focus on the most relevant parts of the
  uploaded documents instead of sending whole files to the model every time.
- RAG also lets me reuse document chunks across many future queries.


Q2. Why not rely only on the LLM for extraction?
A.
- PDF text is noisy and tender/resume formats vary a lot.
- Heuristics give stable low-cost extraction for obvious patterns like names,
  dates, skills, and experience labels.
- The LLM is used as a second pass to fill gaps and keep output schema-shaped.


Q3. Why do you store both FAISS and SQL data?
A.
- FAISS gives fast vector similarity search.
- SQLAlchemy/SQLite gives durable metadata and structured outputs.
- The combination lets the app rebuild FAISS if vector files are deleted and also
  makes document inspection and evidence persistence easier.


Q4. How does duplicate upload handling work?
A.
- On upload, I hash raw bytes with SHA256.
- For the same document type, if that hash already exists and the prior record is
  not failed, I reuse stored chunks, structured data, and evidence instead of
  parsing the file again.


Q5. How do you extract text from difficult PDFs?
A.
- I use a fallback chain:
  PyMuPDF4LLM -> PyMuPDF -> pdfplumber -> Docling -> OCR with Tesseract.
- That improves robustness for digitally generated PDFs and scanned documents.


Q6. How is resume ranking calculated?
A.
- It is deterministic, not LLM-scored.
- Required skill coverage contributes up to 70 points.
- Preferred skill coverage contributes up to 10.
- Role match adds 10.
- Domain match adds 10.
- Experience match adds 10.
- Final score is capped at 100 and then mapped to a verdict.


Q7. What does LangGraph do in this project?
A.
- Right now it is a lightweight reasoning wrapper.
- After deterministic scoring, a single-node LangGraph step adds candidate
  explanations and a summary such as top candidate and shortlist.


Q8. How do you avoid answering from the wrong document?
A.
- The frontend sends active document IDs with every query.
- The backend can restrict retrieval and reasoning to only those active uploads.
- If no explicit active scope is given, it falls back to the latest stored docs.


Q9. What are the biggest limitations of the current version?
A.
- LLM provider is effectively Ollama-only right now.
- Some extraction logic is still heuristic-heavy and format-sensitive.
- Semantic sections are page-local.
- Matching uses simple deterministic scoring instead of deeper competency models.
- There are some stale files and placeholder modules in the repo.


Q10. If you had more time, what would you improve first?
A.
- Add tests around extraction and scoring.
- Improve section detection across page boundaries.
- Add better normalization for skill taxonomies.
- Support PostgreSQL/PGVector for larger scale.
- Add stronger provenance and page-level evidence in the UI.

13. LIKELY WEAK SPOTS AN INTERVIEWER MAY NOTICE

Be ready to answer these honestly.

1. Why is `LLM_PROVIDER` present if Gemini is not implemented?
- Good answer: the docs and env shape were prepared for multi-provider support,

  the abstraction or remove stale configuration to reduce confusion.

  Matching currently leans on resume search driven by extracted tender fields.
  A reasonable next step is to test hybrid retrieval for matching too.

3. Why is LangGraph minimal?
- Good answer: I kept reasoning deterministic and lightweight first. The graph is
  currently a wrapper to keep future extension easy without overcomplicating the
  MVP.

4. Why use SQLite if roadmap mentions PostgreSQL?
- Good answer: SQLite keeps MVP setup simple. The data model already uses
  SQLAlchemy, so moving to PostgreSQL later is easier.

5. Why are some files empty?
- Good answer: they are placeholders from earlier or planned architecture. The
  active runtime path is concentrated in services, RAG, extraction, and UI
  components.
